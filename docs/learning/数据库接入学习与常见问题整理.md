# 数据库接入学习与常见问题整理（基于本次沟通）

本文档汇总我们之前关于“项目接数据库”的讨论，重点解释你提到的不理解点，并给出可执行步骤。

适用对象：
- 当前使用 Flask 项目（如本项目）
- 计划从“本地/内存状态”升级到“数据库持久化”

---

## 1. 先建立整体认知：项目从本地到公网是怎么跑起来的

### 1.1 本地开发 vs 线上部署
- 本地：应用运行在 `localhost:端口`，通常只有你自己可访问。
- 线上：应用运行在公网可访问的计算资源上（云服务器或托管平台），用户通过域名访问。

### 1.2 Render 这种方式本质是什么
- Render 是托管平台，底层仍是服务器资源。
- 你不用自己管很多运维细节（系统、进程守护、证书、部分网络配置）。

### 1.3 大网站（B站/知乎/Reddit）本质是否一样
- 本质一样：代码运行在服务器，用户通过公网访问。
- 差别是规模：大网站通常是多机房、多服务、负载均衡、缓存、队列、数据库集群。

## 2. 服务器到底是什么

你之前的理解是对的：服务器可以理解为“可被远程使用的计算机资源”。

更完整可分 6 层：
1. 硬件层：CPU/内存/磁盘/网卡
2. 虚拟化层：云主机/容器实例
3. 运行层：操作系统 + 运行时（Python/Node）
4. 服务层：Web/API/数据库/缓存
5. 网络层：IP/端口/域名/DNS/HTTPS
6. 运维层：监控/日志/备份/告警/容灾

常见语境中的“服务器”有三种含义：
- 物理机（硬件）
- 云主机（资源单元）
- Server 端程序（软件角色）

## 3. 前端、后端、Flask、JS、数据库之间是什么关系

### 3.1 Flask 是什么
- Flask 是 Python Web 框架。
- 在本项目中，Flask 是后端应用服务，负责接请求、执行业务、回响应。

### 3.2 Flask 和 JS 的关系
- JS 运行在浏览器，是前端交互逻辑。
- JS 通过 `fetch` 调 Flask API。
- Flask 处理后返回 JSON 或文件，JS 再渲染页面。

### 3.3 谁负责从数据库取数据、谁负责展示
- 从数据库取数据：后端（Flask / app.py）
- 业务计算（如对账）：后端
- 页面展示：前端 JS

一句话链路：

```text
前端 JS -> Flask API -> 数据库查询/写入 -> Flask计算 -> JSON响应 -> 前端渲染
```

## 4. “通过 API 传输结果”到底是什么意思

这是你重点问到的不理解点。

API 传输结果，本质是 HTTP 请求/响应的数据交换：

1. 前端发请求（URL + 方法 + 参数）
2. 后端处理业务
3. 后端返回响应（状态码 + JSON/文件）
4. 前端拿响应更新页面

示例：
- 前端发 `POST /api/reconcile`
- 后端执行对账
- 返回：`{ success: true, result: {...} }`
- 前端把 `result` 渲染成汇总卡片和差异表格

结论：
- 前后端不是共享内存，不是互相直接调用函数
- 是通过 HTTP + API 契约传输数据

## 5. 框架是什么，不是什么

### 5.1 你的理解（函数复用）是对的，但需要补充
- 框架不只是“很多函数集合”。
- 框架还包含：
  - 应用生命周期
  - 请求路由机制
  - 统一约定
  - 插件生态

### 5.2 有前后端框架后，开发在写什么
- 业务逻辑（核心价值）
- 数据模型与数据流
- 接口契约
- 页面交互逻辑
- 工程保障（鉴权、日志、测试、部署）

不是“只有业务逻辑重要”，而是这些层共同决定系统能否稳定上线。

## 6. 回到你的项目：接 MySQL 到底要做什么

当前项目特点：
- 使用 Flask + Pandas
- 有 `current_task` 这类进程内状态
- 重启后状态可能丢失，不利于并发与追溯

接数据库的目标：
- 任务、上传记录、对账结果持久化
- 支持按 `task_id` 查询历史与复算

### 6.1 前提条件（必须满足）
1. 有可用 MySQL 实例（本地或云托管）
2. 明确连接参数（host/port/db/user/password）
3. 环境变量管理到位（不能把密码写死在代码）
4. 明确“哪些数据进库，哪些保留文件存储”

### 6.2 需要修改的内容
1. 配置层
- 增加数据库连接配置（`DATABASE_URL` 或等价）

2. 数据层
- 引入 ORM（例如 SQLAlchemy）
- 定义表模型
- 增加迁移机制（Alembic/Flask-Migrate）

3. 业务层
- 用数据库替代 `current_task` 内存状态
- 上传、对账、导出改为按 `task_id` 读写

4. 接口层
- 新增/调整任务接口
- 前端每次请求携带 `task_id`

### 6.3 需要新增的对象（最小可行）
建议先有 3 张核心表：
1. `reconcile_task`：任务主表（状态、文件引用、时间）
2. `upload_record`：上传记录（类型、原文件名、保存路径、解析元信息）
3. `reconcile_result`：对账结果（summary 和差异明细 JSON）

## 7. 推荐实施步骤（按风险从低到高）

1. 接通数据库连接（先验证连通）
2. 建立基础模型并迁移成功
3. 先把“任务主记录”入库
4. 上传接口改为写 `upload_record`
5. 对账接口改为写 `reconcile_result`
6. 前端接入 `task_id` 全链路
7. 回归测试：重启后数据不丢、历史可查、导出一致

建议节奏：每一步都可回退，不要一次性大改。

## 9. 常见误区与风险提醒

1. 误区：接了数据库就“自动稳定”
- 实际还需要事务、索引、异常处理和回归测试。

2. 误区：前端直接连数据库
- 常规架构不这样做，数据库应由后端统一访问。

3. 误区：把所有内容都塞进数据库
- 原始大文件通常存文件系统或对象存储，数据库存元数据和结果更合理。

4. 风险：一次性大改
- 容易导致接口、前端、导出同时出问题。应分步改造。

---

## 10. 你可以用来验收“是否接库成功”的标准

满足以下条件，说明数据库化改造达标：

1. 服务重启后任务和结果仍可查询
2. 支持多个任务并发，不互相覆盖
3. 可按 `task_id` 复查历史结果
4. 页面统计与导出数据一致
5. 异常时有可读日志，能快速定位问题

---

## 11. 一句话总复盘

你这个项目接数据库，不是“换一个存储位置”这么简单，
而是把系统从“临时运行”升级为“可持久、可追踪、可并发、可运维”的产品形态。
