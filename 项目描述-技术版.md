# 自动对账系统项目描述（技术版）

## 1. 项目概述
本项目是一个基于 Flask + Pandas + 原生前端（HTML/CSS/JavaScript）的对账工具，用于对比“供应商对账单”和“系统入库单”，自动输出匹配结果与差异分类。

- 后端入口：`app.py`
- 前端页面：`templates/index.html`
- 前端逻辑：`static/js/app.js`
- 样式：`static/css/style.css`

系统主流程：上传两份 Excel -> 自动识别与字段映射 -> 执行对账 -> 展示差异 -> 导出结果 Excel。

## 2. 前端代码的实现方式

前端采用单页工作台模式，技术实现为“Flask 模板渲染 + 原生 JS 事件驱动 + Fetch API 调后端接口”。

### 2.1 页面结构
- `index.html` 定义侧边栏、上传区、字段映射区、容差配置抽屉、结果 Tab、导出按钮。
- 页面只有一个主路由 `/`，所有业务动作通过异步接口完成，不做前端路由拆分。

### 2.2 状态管理与交互
- 通过全局变量维护会话状态：`vendorData`、`internalData`、`reconcileResult`。
- `DOMContentLoaded` 时初始化导航、主题、上传、Tab、对账按钮、导出、容差抽屉和使用说明弹窗。
- 上传完成后即时渲染文件信息、预览表格，并根据后端建议自动回填供应商映射字段。

### 2.3 前后端通信
- 上传：`POST /api/upload/vendor`、`POST /api/upload/internal`（`FormData` 文件流）。
- 对账：`POST /api/reconcile`（JSON，含字段映射与容差参数）。
- 导出：`POST /api/export`，前端接收二进制 `blob` 并触发浏览器下载。

### 2.4 展示逻辑
- 对账完成后展示汇总 KPI、完全匹配（行级/分组）与各类差异明细。
- 差异区按“仅供应商、仅入库单、数量差异、单价差异、金额差异”分别渲染，支持展开/收起。

## 3. 运行代码的逻辑

## a) 供应商对账单表头定位方法
实现函数：`detect_vendor_table_and_mapping(raw_df, vendor_hint)`。

定位逻辑：
1. 扫描前 `min(len(raw_df), 200)` 行。
2. 对每一行统计：
   - `po_hits`：命中 PO 正则（如 `LY20...`、`PO...`）数量。
   - `item_hits`：命中物料编码正则数量。
   - `num_hits`：可转数字字段数量。
3. 首个满足 `po_hits >= 1 && item_hits >= 1 && num_hits >= 2` 的行作为数据起始行 `data_start`。
4. 表头行取 `header_row = data_start - 1`（最小为 0），该行用于构造列名；空表头自动补 `列1/列2...`，重名自动去重。

补充：若未识别成功，`/api/upload/vendor` 会退回固定方案 `pd.read_excel(..., header=4)` 作为兜底。

## b) 供应商对账单表尾定位方法
同样在 `detect_vendor_table_and_mapping` 内完成，采用“明细模式 + 终止条件”裁剪：

1. 先基于样本列推断 `po_no/item_code/qty/unit_price/amount` 列。
2. 行级识别明细行 `is_detail`：
   - `po_ok`（PO 命中）
   - `item_ok`（物料命中）
   - 数值列命中数 `numeric_hits >= 2`
3. 表尾终止条件：
   - 已进入明细区后，若行中出现 `FOOTER_KEYWORDS`（如“合计/总计/TOTAL”）立即停止。
   - 或已进入明细区后连续 `3` 行非明细，也停止。
4. 最终仅保留 `detail_indices` 对应行，避免将合计、签字、备注等表尾内容纳入对账。

## c) 系统入库单表头定位方法
实现位置：`upload_internal_file()`。

该数据源按固定格式处理，不做启发式表头搜索：
- 使用 `pd.read_excel(filepath)` 直接读取，默认第一行为表头。
- 列名统一转字符串后进入映射与标准化流程。

## d) 系统入库单表尾定位方法
实现位置：`upload_internal_file()`。

当前实现改为 `detect_internal_footer_cutoff(df)` 的“8条规则+防护策略”组合识别，按优先级执行：

1. 强关键词规则（`合计/总计/小计/累计/TOTAL/SUM/Grand Total`）。
2. 首列语义规则（首列出现合计、备注、签字、审核等管理性文本）。
3. 明细结构断裂规则（进入明细后连续非明细行达到阈值）。
4. 空白密度规则（非空占比过低且连续出现）。
5. 字段类型漂移规则（数值结构突然消失，转为文本说明块）。
6. 页脚/分页标识规则（`第X页共Y页`、`Page X of Y`、打印时间等）。
7. 重复表头规则（多页拼接时再次出现列头）。
8. 金额校验辅助规则（疑似合计行金额与前文累计金额近似）。

防护策略：
- 先识别到“明细起始”后才允许截断。
- 设置最小明细行保护（当前 `min_detail_rows=5`）。
- 弱关键词不能单独触发，需叠加结构断裂等信号。
- 软规则采用评分阈值触发（`soft_score >= 3`）。

接口会返回 `footer_detection` 元数据（是否命中、命中原因、截断行号），用于排查和审计。

## e) 对账单的方法逻辑
核心函数：`perform_reconciliation(...)`。

### e.1 数据标准化
- 供应商侧：`standardize_vendor_data`，完成列映射、数量转整型、单价保留 4 位、金额数值化、物料编码清洗。
- 系统侧：`standardize_internal_data`，完成同类标准化。

### e.2 统一匹配键
- 供应商：`po_item_key = po_no + '|' + item_code_clean`
- 系统：`po_item_key = po_no + '|' + item_code`

### e.3 分组与分类优先级
按 `po_item_key` 聚合并逐键比对，分类顺序是：
1. `差异项`：某一侧完全不存在该 key。
2. `数量差异`：key 都存在，但汇总数量差超容差。
3. `单价差异`：数量可对齐，但无法按“数量+单价”完成配对。
4. `金额差异`：数量和单价可配对，但金额差超容差。
5. `完全匹配`：行级严格配对后双方无剩余。

### e.4 行级配对机制
- 严格配对条件：`qty`、`unit_price`、`amount` 均在容差内。
- 使用 `_pair_rows` 做一对一贪心配对，输出匹配与残差。
- 数量差异场景额外使用 `split_qty_residual_rows`，优先“数量+单价”再“仅数量”配对，提取真正残差行供前端展示。

### e.5 输出结果
返回结构包含：
- `summary` 统计
- `matched_pairs`（行级匹配）
- `matched_groups` / `matched_list`（分组视图）
- `diff_items/diff_qty/diff_price/diff_amount` 四类差异

## f) 数据导入与导出采用的方法和逻辑

### f.1 数据导入
接口：`/api/upload/vendor`、`/api/upload/internal`。

- 前端通过 `FormData` 上传 Excel。
- 后端做扩展名与大小校验，文件落盘到 `uploads/`。
- 供应商文件：先按内容模式自动定位表格并建议字段映射；失败时走固定表头兜底。
- 系统文件：固定结构读取后，采用 8 规则组合识别并裁剪表尾。
- 两侧均返回：原文件名、行数、列名、前 10 行预览（含 NaN/日期清洗）；系统文件额外返回 `footer_detection` 信息。

### f.2 数据导出
接口：`/api/export`。

- 使用 `io.BytesIO` + `pd.ExcelWriter(engine='openpyxl')` 内存生成工作簿。
- 默认导出多 Sheet：`汇总`、`完全匹配`、`差异项`、`数量差异`、`单价差异`、`金额差异`（按实际有无数据写入）。
- 通过 `send_file` 以附件流返回，文件名形如 `对账结果_YYYYMMDD_HHMMSS.xlsx`。

## 4. 技术特征与注意点
- 采用模板+规则混合识别供应商表结构，兼顾灵活性与稳定性。
- 通过容差参数控制匹配灵敏度（价格/数量/金额）。
- 结果对象在返回前执行 `clean_dict`，将 NaN/Inf 转为 JSON 友好值。
- 当前任务数据保存在进程内全局变量 `current_task`，适合单实例轻量场景；如需多用户并发，建议迁移至数据库或任务存储层。
